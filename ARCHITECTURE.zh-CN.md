# NG Manager 架构约束

## 文档目的

本文档用于定义 `ng-manager` 当前的架构约束，确保产品在支持两个用户入口的同时，仍然保持统一的运行模型。

## 产品模型

`ng-manager` 采用“双入口、单后端、单前端”的模型：

- `cli` 是轻量入口，面向不希望安装桌面端的用户。
- `desktop` 是桌面壳入口，面向希望使用本地安装应用的用户。
- `server` 是两个入口共用的唯一本地后端。
- `webapp` 是两个入口共用的唯一 UI。

系统应始终被视为“同一个产品的两种启动方式”，而不是两套独立产品。

## 主运行链路

规范的运行链路如下：

1. 用户启动 `cli` 或 `desktop`。
2. 入口层确保本地 `server` 可用。
3. `server` 提供本地 API，并承载 UI 运行所需能力。
4. 用户通过 `webapp` 进行交互。

无论通过哪种入口启动，这条链路都应保持一致。

## 组件职责

### `packages/cli`

`cli` 负责：

- 启动或复用本地 `server`
- 停止本地 `server`
- 检查本地 `server` 状态
- 在需要时打开浏览器访问 UI

`cli` 不应承载业务逻辑、API 领域逻辑或 UI 逻辑。

### `desktop`

`desktop` 负责：

- 应用窗口生命周期
- 本地运行进程生命周期
- 托盘、系统对话框、系统集成等桌面壳能力

`desktop` 不应重复实现服务端业务逻辑，也不应维护独立的 API 实现。

### `packages/server`

`server` 是唯一后端运行时，负责：

- 提供本地 HTTP API
- 承载本地系统能力与文件编排能力
- 在生产模式下提供前端静态资源
- 暴露入口层所需的健康检查与生命周期接口

`server` 是本地运行状态的唯一后端事实来源。

### `webapp`

`webapp` 是唯一前端，负责：

- 应用界面
- 用户交互流程
- 展示 `server` 返回的状态与数据

`webapp` 应通过运行时配置适配不同启动模式，而不是分别维护 CLI 模式页面和桌面模式页面。

## 共享运行规则

### 单一本地后端

在同一个用户数据目录下，正常情况下应只存在一个可复用的本地 `server` 实例，除非有明确理由需要隔离实例。

`cli` 和 `desktop` 都应优先遵循以下策略：

1. 探测现有健康的 `server`
2. 若可用则复用
3. 仅在必要时启动新实例

两种入口不应采用互相冲突的启动策略。

### 统一服务契约

两个入口都必须基于同一份后端契约来完成：

- 健康检查
- 启动完成判断
- 关闭服务
- 服务地址发现

健康检查必须依据 `server` 的返回契约，而不是入口层各自的私有判断。

### 统一启动编排

以下能力应沉到共享代码中，而不是分别写在 `cli` 和 `desktop` 中：

- 端口选择
- lock 文件处理
- 健康轮询
- 启动超时处理
- 优雅关闭

## 前端运行模式

`webapp` 可以运行在不同环境中，但它始终是同一个前端。

浏览器启动和桌面端启动之间的差异，应通过运行时配置表达，例如：

- 运行模式
- 后端服务地址
- 是否具备桌面壳能力

功能差异应通过能力开关控制，而不是通过分叉页面或重复模块实现。

## 仓库角色划分

仓库中各目录的角色如下：

- `packages/*`：可复用的库与后端包
- `webapp`：主前端应用
- `desktop`：桌面壳应用
- `ws-lab`：非主线实验工作区
- `test-ngm`：本地示例或联调工程
- `scripts`：仓库维护脚本

只有 `cli`、`desktop`、`server`、`webapp` 共同构成主产品运行链路。

## 明确不做的事情

当前架构明确避免以下情况：

- 为 CLI 和桌面端分别维护两套后端实现
- 为 Web 和桌面模式分别维护两套前端实现
- 在入口层写业务规则
- 让 `cli` 和 `desktop` 的启动逻辑长期分叉

## 变更原则

新增功能时，遵循以下规则：

1. 先明确该行为属于哪一层
2. 优先复用共享运行服务，避免重复编排
3. 保持入口层足够薄
4. 保持 `server` 作为唯一后端事实来源
5. 保持 `webapp` 作为唯一 UI 事实来源

任何破坏“双入口、单后端、单前端”模型的改动，都应视为架构例外，并被明确评审。
