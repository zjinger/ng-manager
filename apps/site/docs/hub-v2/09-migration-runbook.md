# Hub v1 -> v2 迁移执行 Runbook

最后更新：2026-03-23

## 1. 文档目的

本文档用于把 [08 数据迁移映射清单](/hub-v2/08-migration-mapping) 落成可执行迁移手册，重点解决：

- 迁移前需要准备什么
- 迁移当天按什么顺序执行
- 每个阶段怎么验证
- 灰度切流怎么做
- 出现问题时如何回滚

本文档是迁移演练和正式切换的操作基线，不是设计说明。

---

## 2. 适用范围

适用于以下场景：

- `apps/hub` 已经在线运行
- `apps/hub-v2` 已完成基础功能并可独立部署
- 准备把 v1 数据导入 v2
- 准备按模块逐步把访问入口切到 v2

不适用于以下场景：

- v2 仍处于不可启动或不可登录状态
- v2 主流程尚未通过基础回归
- 希望让 v1 和 v2 长期双写同一业务域

## 4. 角色与职责

### 4.1 技术负责人

负责：

- 冻结迁移窗口
- 审批切流顺序
- 决定是否回滚

### 4.2 服务端负责人

负责：

- 迁移脚本执行
- 数据校验
- v2 服务部署
- 切流后问题排查

### 4.3 前端负责人

负责：

- v2 页面路由与入口切换
- 关键页面回归验证
- 通知、Dashboard、Issue、RD 页面功能检查

### 4.4 QA / 业务验收人

负责：

- 登录、项目切换、Issue、RD、内容管理主流程验收
- 抽样数据核对

### 4.5 运维/发布负责人

负责：

- 备份源库
- 发布新版本
- 回滚入口切换

## 5. 前置条件

正式迁移前，以下条件必须全部满足：

- `apps/hub-v2/server` 构建通过
- `apps/hub-v2/web` 构建通过
- v2 数据库 migration 已全部执行成功
- v2 最小 smoke test 已通过
- 迁移脚本已在演练库至少成功执行 1 次
- 已完成字段映射和状态映射评审
- 已明确切流顺序和业务冻结窗口
- 已准备回滚入口方案

若任一条件不满足，不应进入正式迁移窗口。

---

## 6. 环境准备

### 6.1 源库准备

准备 v1 运行中的 SQLite 数据库文件：

- 记录路径
- 确认文件可读
- 记录当前文件大小和修改时间

### 6.2 目标库准备

准备 v2 目标 SQLite 数据库文件：

- 确认路径
- 确认 `db:migrate` 已执行
- 确认目标库为空库或已按计划重置

### 6.3 备份准备

至少保留以下备份：

- v1 数据库完整备份
- v2 目标库迁移前空库备份
- 当前线上静态前端包或部署产物

### 6.4 发布准备

准备以下发布物：

- v2 server 构建产物
- v2 web 构建产物
- 迁移脚本
- 校验脚本

---

## 7. 迁移执行顺序

推荐执行顺序：

1. 冻结写入窗口
2. 备份 v1 数据库
3. 初始化或重置 v2 目标库
4. 执行基础域迁移
5. 执行内容域迁移
6. 执行 Issue 域迁移
7. 执行 RD 域迁移
8. 执行 Dashboard 偏好迁移
9. 做数量校验和抽样校验
10. 启动 v2 并做 smoke test
11. 按模块灰度切流

不建议：

- 一边迁移一边继续让 v1 写数据
- 所有模块一次性整体切站
- 未校验即切流

---

## 8. 迁移模块步骤

### 8.1 基础域

迁移范围：

- `users`
- `admin_users -> admin_accounts`
- `projects`
- `project_members + project_member_roles -> project_members`

执行要求：

- 优先迁移账号和用户
- 再迁移项目
- 最后重组项目成员

校验点：

- 用户总数一致
- 登录账号总数一致
- 项目总数一致
- 每个项目至少能抽样检查成员数
- `admin_accounts.user_id` 引用合法

### 8.2 内容域

迁移范围：

- `announcements`
- `announcement_reads`
- `documents`
- `releases`
- `shared_config -> shared_configs`
- `uploads`

校验点：

- 公告、文档、发布记录数量一致
- 已读记录数量一致
- 上传记录数量一致
- 抽样检查内容正文、发布时间、项目归属

### 8.3 Issue 域

迁移范围：

- `issues`
- `issue_comments`
- `issue_attachments`
- `issue_participants`
- `issue_action_logs -> issue_logs`

执行要求：

- 先迁 `issues`
- 再迁关联表
- 最后迁日志

校验点：

- Issue 总数一致
- 每条抽样 Issue 的评论数、附件数、参与人数一致
- 状态、责任人、验证人一致
- 日志时间线顺序正确

### 8.4 RD 域

迁移范围：

- `rd_stages`
- `rd_items`
- `rd_logs`

校验点：

- 阶段数量一致
- 研发项数量一致
- 状态和阶段归属一致
- 日志时间线可追溯

### 8.5 Dashboard 偏好

迁移范围：

- `dashboard_preferences`

校验点：

- 偏好数量一致
- 抽样检查统计卡配置 JSON 是否完整

## 9. 验证清单

### 9.1 数量校验

迁移完成后，必须输出以下统计：

- 用户数
- 登录账号数
- 项目数
- 项目成员数
- 公告数
- 文档数
- 发布数
- 共享配置数
- 上传数
- Issue 数
- Issue 评论数
- Issue 附件数
- Issue 协作人数
- Issue 日志数
- RD 阶段数
- RD 项数
- RD 日志数
- Dashboard 偏好数

### 9.2 外键校验

至少检查：

- `admin_accounts.user_id -> users.id`
- `project_members.project_id -> projects.id`
- `project_members.user_id -> users.id`
- `issues.project_id -> projects.id`
- `issues.assignee_id / reporter_id / verifier_id -> users.id`
- `issue_attachments.upload_id -> uploads.id`
- `rd_items.stage_id -> rd_stages.id`

### 9.3 抽样校验

建议抽样：

- 20 条 Issue
- 10 条 RD
- 10 个项目成员关系
- 10 条公告已读记录

抽查内容：

- 标题
- 状态
- 责任人
- 项目归属
- 评论数
- 日志数
- 时间字段

### 9.4 行为校验

迁移完成后至少执行以下行为检查：

1. 登录 v2
2. 切换项目
3. 查看 Dashboard
4. 打开 Issue 列表与详情
5. 对一条测试 Issue 做评论
6. 打开 RD 列表/看板
7. 打开内容管理页
8. 查看通知中心

## 10. 灰度切流步骤

推荐切流顺序：

1. `users / projects`
2. `content`
3. `issues`
4. `rd`
5. `dashboard / notifications`

执行规则：

- 每次只切一个业务域
- 已切流业务域只允许 v2 写
- 未切流业务域继续由 v1 写
- 每次切流后都要做最小回归

推荐方式：

- 先给内部用户开放 `hub-v2` 入口
- 灰度验证通过后，再把导航入口切换到 v2
- 不直接停掉 v1

---

## 11. 回滚规则

回滚只回流量，不回数据。

即：

- v2 出现严重问题时，把对应业务域入口切回 v1
- 不在故障时尝试把 v2 数据反向导回 v1

回滚触发条件建议：

- 登录或项目切换不可用
- Issue / RD 主流程阻断
- 数据明显错乱
- 抽样数据不一致且无法快速修复

回滚步骤：

1. 暂停继续切流
2. 把入口切回 v1
3. 保留 v2 当前数据库用于排查
4. 记录问题域、问题时间、影响范围
5. 修复后重新演练再切

---

## 12. 演练要求

正式上线前，至少进行一次全量演练。

演练要求：

- 使用接近生产的数据副本
- 走完整迁移顺序
- 走完整校验清单
- 走一次灰度切流演练
- 走一次回滚演练

演练输出物：

- 演练时间
- 使用数据源
- 迁移耗时
- 问题记录
- 修复项清单
- 是否可进入正式上线

---

## 13. 上线当天检查表

### 13.1 开始前

- 确认迁移窗口已通知
- 确认 v1 进入写冻结
- 确认备份完成
- 确认脚本版本与文档版本一致

### 13.2 执行中

- 记录每一步开始时间和结束时间
- 记录每模块迁移数量
- 记录所有 warning / conflict
- 完成数量校验和抽样校验

### 13.3 切流后

- 登录检查
- 项目切换检查
- Dashboard 检查
- Issue 检查
- RD 检查
- 通知检查

### 13.4 收尾

- 记录最终切流时间
- 记录遗留问题
- 记录是否需要二次迁移或补数据

1. `apps/hub-v2/server/src/db/migrate-from-v1.ts`
