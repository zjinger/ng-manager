# 研发项 UI 设计说明

## 1. 设计目标

基于现有 Hub v2 的页面结构与视觉体系，对“研发项”模块进行增量式改版，目标是让研发可以在**尽量少改框架**的前提下，完成以下能力升级：

1. 将单执行人模式升级为**团队成员协作模式**。
2. 将“确认人”统一重命名为**验证人**。
3. 在列表页与详情页补充**实际时间**与**时间状态**展示。
4. 引入**成员个人进度 + 主进度自动合并**的可视化交互。
5. 保持与现有 Hub v2 的导航、卡片、表格、抽屉、弹窗等组件风格一致。

---

## 2. 页面拆分说明

### 2.1 页面一：研发项列表页

### 页面定位
研发项模块的主入口，用于高频浏览、筛选、搜索、批量判断任务状态。

### 页面目标
- 快速查看当前有哪些研发项正在推进。
- 一眼判断研发项是否为多人协作、当前整体进度如何、时间上是否延期。
- 支持从列表直接进入详情页。

### 结构分区

| 区域 | 内容 | 说明 |
|---|---|---|
| 页面标题区 | 标题、说明文案、导出、新建按钮 | 延续现有页面头部结构 |
| 统计卡片区 | 本期研发项、按时完成率、延期中、成员更新率 | 承接 PRD 的效能分析诉求 |
| 筛选工具栏 | Sprint、研发状态、时间状态、我的角色、搜索、列表/看板切换 | 兼容旧页面的使用习惯 |
| 列表表格区 | 研发项核心字段 | 作为主视图 |

### 列表字段建议

| 字段 | 是否保留/新增 | 展示规则 |
|---|---|---|
| 编号 | 保留 | 如 `RDI-031` |
| 标题 | 保留 | 允许一行主标题 + 一行辅助说明 |
| 团队成员 | 新增替换 | 以成员 chip 展示，附带个人进度 |
| 验证人 | 重命名 | 替代“确认人” |
| 整体进度 | 新增强化 | 平均值进度条 + 百分比 |
| 实际开始 | 新增 | 无值显示 `-` |
| 实际结束 | 新增 | 无值显示 `-` |
| 工期 | 新增 | 已开始则显示实际工期 |
| 时间状态 | 新增 | 未开始 / 进行中 / 提前 / 按时 / 延期 |
| 研发状态 | 保留 | todo / doing / blocked / done |

### 列表页交互说明
- 点击整行进入详情抽屉。
- 点击成员进度 chip 可作为后续扩展入口，用于快速更新个人进度。
- 列表视图适合信息密度场景；默认进入列表视图。

---

### 2.2 页面二：研发项看板页

### 页面定位
适合按状态拖拽、快速浏览任务推进阶段。

### 页面目标
- 帮助 PM 或研发负责人从宏观上查看待办、进行中、阻塞、已完成分布。
- 在卡片层面体现多人协作与整体进度，而不展开全部细节。

### 结构分区

| 区域 | 内容 | 说明 |
|---|---|---|
| 看板列 | 待办 / 进行中 / 阻塞 / 已完成 | 沿用旧稿结构 |
| 卡片头部 | 编号、标题 | 核心识别信息 |
| 卡片状态区 | 研发状态标签 | 与列状态保持一致 |
| 卡片成员区 | 团队成员简写进度 | 只展示关键成员与进度 |
| 卡片底部 | 时间状态 / 整体进度 | 辅助判断效率 |

### 看板卡片设计重点
- 不再展示单执行人，而是展示**多成员进度概览**。
- 时间状态与整体进度同时出现，避免只看到“doing”但无法判断是否存在延期风险。
- 卡片点击进入详情抽屉，保证信息承接一致。

---

### 2.3 页面三：研发项详情抽屉

### 页面定位
研发项的核心工作台，是此次改版最重要的页面。

### 页面目标
- 让成员看到自己和其他人的进度。
- 让 PM 看到是否具备“申请完成”的条件。
- 让项目管理者看到时间信息与状态流转。

### 抽屉布局
采用 **左主右辅** 双栏结构：

| 区域 | 内容 | 设计目的 |
|---|---|---|
| 左侧主内容区 | 整体进度、成员进度、研发描述、进度历史 | 承载主要操作与理解路径 |
| 右侧属性区 | 操作按钮、基础属性、时间信息 | 承载状态操作与辅助信息 |

### 左侧模块拆分

#### A. 整体进度模块
- 展示整体进度百分比与大进度条。
- 说明主进度计算逻辑：默认按成员平均值。
- 提示完成规则：
  - 全员进度 ≥ 80%：可申请完成
  - 全员进度 = 100%：提示可完成

#### B. 成员进度模块
- 成员头像、姓名、角色、最后更新时间、进度说明。
- 每个成员一行：
  - 左侧：身份信息
  - 中间：进度条
  - 右侧：百分比 + 更新按钮
- 团队成员统一平级，不再区分“主执行人 / 参与人”。

#### C. 研发描述模块
- 继续展示原有描述内容。
- 用于承接业务上下文、开发范围、联调说明。

#### D. 进度历史模块
- 记录成员进度变化与系统时间记录。
- 用于追溯谁在什么时间更新了进度。

### 右侧模块拆分

#### A. 操作模块
- 申请完成
- 标记阻塞
- 继续开发
- 管理员强制完成

#### B. 基本属性模块
- 创建人
- 团队成员数量
- 验证人
- 优先级
- 当前状态

#### C. 时间信息模块
- 计划开始
- 计划结束
- 实际开始
- 实际结束
- 计划工期
- 实际工期
- 时间状态提示

### 详情页交互重点
- 当前用户若是团队成员，可点击“更新”进入进度更新弹窗。
- 若未满足完成门槛，操作区要明确提示原因，避免误操作。
- 时间信息区应在已开始后自动显示工期计算结果。

---

### 2.4 页面四：新建 / 编辑研发项弹窗

### 页面定位
用于创建研发项与修改基础信息。

### 页面目标
- 从表单层面完成字段语义升级。
- 保证研发录入模型与展示模型一致。

### 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 研发标题 | 文本 | 必填 |
| 阶段 | 下拉 | Sprint / 版本 |
| 优先级 | 下拉 | P0 / P1 / P2 |
| 团队成员 | 多选 | 至少 1 人 |
| 验证人 | 单选 | 替代确认人 |
| 创建人 | 只读或默认当前人 | 由系统带入 |
| 计划开始 | 日期 | 计划字段 |
| 计划结束 | 日期 | 计划字段 |
| 研发描述 | 多行文本 | 说明工作内容 |

### 设计重点
- “团队成员”使用标签化多选交互，方便增删。
- “验证人”独立展示，避免和成员列表混淆。
- 不在表单中要求填写实际时间，保持与 PRD 中“自动记录”一致。

---

### 2.5 页面五：更新我的进度弹窗

### 页面定位
成员的高频轻操作入口。

### 页面目标
- 让成员快速更新自己的进度。
- 同时补充本次进展说明，便于他人理解。

### 结构说明

| 区域 | 内容 |
|---|---|
| 成员信息 | 当前更新人 |
| 进度值 | 滑杆 + 实时百分比 |
| 进度说明 | 多行输入，选填 |
| 底部按钮 | 取消 / 保存进度 |

### 交互规则
- 默认只能更新“我的进度”。
- 管理员可扩展支持代更新他人进度。
- 保存后需同步刷新：
  - 当前成员进度
  - 整体进度
  - 进度历史

## 4. 状态说明文档

### 4.1 研发状态

| 状态 | 标识 | 含义 | 页面表现 |
|---|---|---|---|
| 待办 | todo | 尚未开始 | 灰色标签 |
| 进行中 | doing | 已开始开发 | 蓝色标签 |
| 阻塞 | blocked | 因依赖/异常无法推进 | 红色标签 |
| 已完成 | done | 研发完成，待后续闭环 | 绿色标签 |

### 4.2 时间状态

| 状态 | 触发条件 | 页面表现 |
|---|---|---|
| 未开始 | `actual_start_at` 为空 | 不强调 |
| 进行中 | 已开始但未结束 | 黄色/橙色提示 |
| 提前完成 | `actual_end_at < plan_end_at` | 绿色标签 |
| 按时完成 | `actual_end_at = plan_end_at` | 蓝色标签 |
| 延期完成 | `actual_end_at > plan_end_at` | 红色标签 |

### 4.3 完成规则状态

| 状态 | 判定方式 | 页面提示 |
|---|---|---|
| 不可申请完成 | 存在成员进度 < 80% | 操作区提示具体成员 |
| 可申请完成 | 全员进度 ≥ 80% | 申请完成按钮高亮 |
| 可自动提示完成 | 全员进度 = 100% | 顶部提示“可完成” |
| 强制完成 | 管理员执行 | 需填写原因 |

### 4.4 角色权限状态

| 操作 | 团队成员 | 验证人 | 项目管理员 |
|---|---|---|---|
| 查看详情 | 是 | 是 | 是 |
| 评论 | 是 | 是 | 是 |
| 更新自己的进度 | 是 | 否 | 是 |
| 更新他人进度 | 否 | 否 | 是 |
| 申请完成 | 是（满足规则时） | 否 | 是 |
| 强制完成 | 否 | 否 | 是 |
| 编辑基础信息 | 仅创建人 | 否 | 是 |
| 删除 | 仅创建人或管理员 | 否 | 是 |

## 5. 研发实现建议

### 5.1 前端拆分建议

| 模块 | 建议组件名 |
|---|---|
| 列表页 | `RdItemListPage` |
| 看板页 | `RdItemBoardView` |
| 详情抽屉 | `RdItemDetailDrawer` |
| 新建/编辑弹窗 | `RdItemFormModal` |
| 更新进度弹窗 | `RdItemProgressModal` |
| 成员进度组件 | `RdMemberProgressList` |
| 时间信息组件 | `RdTimeInfoCard` |
| 时间状态标签 | `RdTimeStatusBadge` |

### 5.2 渲染顺序建议
1. 先完成字段替换：确认人 → 验证人。
2. 再完成表单升级：单执行人 → 团队成员。
3. 然后补列表字段：时间与整体进度。
4. 最后完成详情页：成员进度、时间信息、进度更新弹窗。

### 5.3 兼容性建议
- 在旧数据仍为单执行人的情况下，前端可将单人数据映射为 1 个团队成员。
- 若 `actual_start_at` / `actual_end_at` 为空，则统一显示 `-`。
- 若个人进度不存在，则按 `0%` 展示并提示“未更新”。

## 6. 交付结论

这次研发项 UI 改版不是完全推翻旧页面，而是在现有 Hub v2 结构上做**低风险、可渐进上线**的增强。

从交付角度看，建议研发按下面顺序落地：

1. **先改表单字段与详情字段语义**，完成“验证人”和“团队成员”替换。
2. **再补列表与详情时间展示**，完成实际时间可视化。
3. **最后补个人进度体系**，完成真正的多人协作闭环。

这样既能尽快满足 PRD，也能最大限度复用已有页面和样式体系。
