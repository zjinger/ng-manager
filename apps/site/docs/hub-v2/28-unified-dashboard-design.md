# Hub-v2 单工作台融合与可配置方案

> 来源：由 `apps/hub-v2/server/docs/unified-dashboard-design.md` 迁入站点文档。本文说明统一 `/dashboard` 的产品与实现口径，配合 [22 当前实现总览](/hub-v2/22-current-implementation-overview) 与 [23 财务与审批总览](/hub-v2/23-finance-approval-overview) 阅读。

## Summary

当前 `hub-v2` 同时存在协作工作台 `/dashboard` 与报销工作台 `/re-dashboard` 两个首页入口，导航中也分别存在“工作台”和“财务中心 -> 报销工作台”两套首页概念。这会带来以下问题：

- 首页心智重复，用户难以判断“默认应该去哪个工作台”
- 报销职责用户登录后会看到大量与自己无关的协作入口
- 同时具备协作职责和报销职责的用户，需要在两个工作台之间来回跳转
- 顶部返回入口、默认重定向、权限拦截 fallback 目前都以 `/dashboard` 为主，与报销侧入口不一致

本方案将 `hub-v2` 收敛为**单一路由智能首页**：只保留一个 `/dashboard` 作为统一工作台入口。首页采用“**报销优先混合首页**”，并按当前登录用户的权限动态显示报销区块、协作区块或两者混合。同时为所有登录用户提供“**卡片显示/隐藏 + 排序**”的个性化能力，且配置以**服务端持久化**方式保存，保证跨浏览器和跨设备一致。

## Current State

### 现有入口

- 协作工作台主入口：`/dashboard`
- 报销工作台主入口：`/re-dashboard`
- 左侧导航中：
  - `工作台` 分组下有 `/dashboard`
  - `财务中心` 分组下有 `/re-dashboard`

### 现有问题

1. **双首页入口冲突**
   - 登录后默认仍以 `/dashboard` 为主
   - 报销侧功能又单独引入了 `/re-dashboard`
   - 用户很容易把“报销工作台”误解为另一个系统首页

2. **导航信息噪音**
   - 纯报销职责用户不需要看到测试跟踪、研发管理、内容管理、项目管理等协作菜单
   - 当前导航结构对这类用户不够收敛

3. **混合职责用户效率低**
   - 同时具备成员角色、报销管理员、财务、出纳等职责的用户，需要在两个工作台之间切换
   - 首页卡片内容无法按个人关注重点调整

4. **入口与权限不一致**
   - 顶部“返回工作台”、品牌入口、若干 guard fallback 都默认回 `/dashboard`
   - 报销侧首页没有成为系统级统一入口

## Target Design

### 1. 单一路由智能首页

- `/dashboard` 成为唯一默认首页入口
- `/re-dashboard` 不再作为主入口，改为兼容重定向到 `/dashboard`
- 顶部“返回工作台”、品牌入口、默认跳转、权限守卫 fallback 全部统一指向 `/dashboard`

### 2. 首页内容按用户能力动态分流

首页使用统一容器页，按用户能力渲染不同内容：

#### 纯报销职责用户

仅显示报销域相关内容：

- 待我审批
- 我提交审批中
- 本月完成数量 / 金额
- 最近待办
- 最近报销单
- 报销公告
- 报销快捷入口

不显示测试跟踪、研发管理、项目公告、最新文档等协作卡片。

#### 纯协作职责用户

仅显示协作域相关内容：

- 我的协作待办
- 我提的测试单 / 研发动态
- 协作公告
- 最新文档
- 当前已有的协作快捷入口与统计卡片

不显示报销审批、报销统计、报销公告等报销卡片。

#### 混合职责用户

同时显示两类内容，但采用“**报销优先混合首页**”布局：

- 报销卡片排在前面，作为主视图
- 协作卡片排在后面，作为次级区块

这样既不丢失协作能力，又避免混合用户首页过于偏协作。

### 3. 导航裁剪

左侧导航按当前用户实际权限动态裁剪，而不是对所有人展示同一套导航。

#### 对只有报销职责的用户

隐藏以下协作域菜单：

- 协作中心
- 内容中心
- 项目管理
- 其他无实际权限的协作入口

保留以下入口：

- 工作台
- 财务中心
- 通知
- 个人中心

#### 对具备协作权限的用户

保留协作菜单，但导航中只保留一个系统首页入口，即 `/dashboard`。

#### 财务中心保留功能入口

财务中心不再承载首页概念，仅保留功能页入口：

- 报销公告管理
- 报销管理
- 我的报销

其中原“报销工作台”菜单项应移除，或改为跳回统一 `/dashboard`，避免继续制造双首页心智。

去掉“待我审批”和“历史报销单”入口，避免与首页重复。
新增“报销管理”页面，针对报销管理员（即 expense_manager 角色）/财务/出纳（即 finance/cashier 角色）开放，展示所有报销单列表和审批入口。
报销管理页面和历史报销单页面类似，历史报销单展示已完成的，报销管理除了展示历史，还展示待审批和审批中的。



### 4. 统一首页实现策略

本期不强制把协作工作台与报销工作台的后端接口聚合为一个新接口，而是先由前端聚合两套现有数据源。

#### 协作域继续复用现有接口

- `/dashboard/home`
- `/dashboard/stats`
- `/dashboard/todos`
- `/dashboard/reported-issues`
- `/dashboard/activities`
- `/dashboard/announcements`
- `/dashboard/documents`

#### 报销域继续复用现有接口

- `/reimbursements/dashboard`
- `/reimbursements/stats`
- 已发布报销公告读取接口

#### 前端新增能力判断

统一首页需要先计算以下能力：

- `canAccessReimbursementWorkspace`
- `canAccessCollaborationWorkspace`
- `isReimbursementOnlyUser`
- `isCollaborationOnlyUser`
- `isMixedWorkspaceUser`

然后按能力决定：

- 是否发起报销域请求
- 是否发起协作域请求
- 渲染哪些卡片

### 5. 本期不做的事情

为控制一期复杂度，以下能力明确不纳入本期：

- 不做首页顶部“协作 / 报销”视图切换
- 不做登录后手动选择进入哪个工作台
- 不做自由拖拽布局
- 不做主区/侧栏自由编排
- 不做列宽或栅格宽度配置
- 不强制新增统一 dashboard 聚合后端接口

## Dashboard Preferences

### 1. 配置目标

为**所有登录用户**提供工作台个性化配置能力，而不是只对混合职责用户开放。

原因：

- 纯报销用户也可能只关心审批待办，不关心统计卡片
- 纯协作用户也可能只关心待办和公告，不关心全部卡片
- 混合职责用户的可配置只是最明显场景，但不是唯一场景

### 2. 本期配置范围

本期只支持两项配置：

1. **显示 / 隐藏卡片**
2. **卡片排序**

不支持拖拽网格或布局分栏，避免把工作量扩散到复杂布局系统。

### 3. 卡片建模方式

每张卡片应有稳定、可持久化的 key。建议使用按业务域分组的 key：

- `reimbursement.todo`
- `reimbursement.claims`
- `reimbursement.stats`
- `reimbursement.announcements`
- `reimbursement.quickAccess`
- `collab.todos`
- `collab.issues`
- `collab.activities`
- `collab.announcements`
- `collab.documents`

配置中只保存：

- `key`
- `visible`
- `order`

不保存接口数据快照，不保存布局尺寸。

### 4. 默认模板

如果用户未保存配置，则使用系统默认模板：

- 纯报销用户：报销卡片默认模板
- 纯协作用户：协作卡片默认模板
- 混合职责用户：报销优先混合模板

默认模板由前端或后端统一定义，避免首次进入首页时出现空白或无序状态。

### 5. 权限过滤规则

配置读取时必须做权限过滤：

- 用户无权限访问的卡片，不返回给前端候选列表
- 历史配置里残留的无权限卡片，在读取时自动剔除
- 保存配置时，后端也需校验卡片 key 是否属于当前用户可配置范围

## API / Data Contract

### 1. 路由兼容

- `/dashboard`：统一首页
- `/re-dashboard`：兼容重定向到 `/dashboard`

### 2. 用户工作台配置存储

建议新增用户级工作台配置持久化表，例如：

- `dashboard_preferences`

最小字段建议：

- `id`
- `user_id`
- `dashboard_code`
- `layout_json`
- `updated_at`

约束建议：

- `user_id + dashboard_code` 唯一

其中：

- `dashboard_code` 首版固定为 `home`
- `layout_json` 保存卡片显示/排序配置

### 3. 配置接口

建议新增接口：

- `GET /api/admin/dashboard/preferences`
- `PUT /api/admin/dashboard/preferences`

#### GET 返回建议

- 当前用户的有效配置
- 若无配置，则返回系统默认模板
- 已经过滤无权限卡片

#### PUT 请求建议

前端提交当前用户可见卡片的配置：

- 卡片 key
- visible
- order

后端负责：

- 校验 key 合法性
- 校验是否属于当前用户可配置范围
- 去重
- 规范化排序

### 4. 前端视图模型

建议新增以下前端模型：

- `WorkspaceCapabilities`
- `DashboardWidgetKey`
- `DashboardWidgetPreference`
- `DashboardPreferences`
- `UnifiedDashboardViewModel`

这些模型用于隔离：

- 路由/导航权限判断
- 配置渲染
- 首页数据组装

## Test Plan

### 路由与入口

- 登录后默认进入 `/dashboard`
- 访问 `/re-dashboard` 会重定向到 `/dashboard`
- 顶部“返回工作台”、品牌入口、权限 fallback 都统一回 `/dashboard`

### 首页内容分流

- 纯报销用户：首页只显示报销卡片
- 纯协作用户：首页只显示协作卡片
- 混合职责用户：首页同时显示两类卡片，且报销卡片默认排在前面

### 导航裁剪

- 纯报销用户看不到协作中心、内容中心、项目管理
- 有协作权限用户仍可见协作菜单
- 财务中心的功能入口保持可用

### 卡片配置

- 用户可隐藏某张卡片并刷新后保持
- 用户可调整卡片顺序并刷新后保持
- 切换浏览器或重新登录后配置仍生效
- 无权限卡片不会被返回或渲染

### 回归

- 报销公告管理、我的报销、待我审批、历史报销单入口不受影响
- 协作内容页、项目页、测试跟踪页对有权限用户仍可访问

## Assumptions

- 只保留一个系统首页 `/dashboard`
- `/re-dashboard` 仅作为兼容重定向保留
- 首页采用“报销优先混合首页”
- 所有登录用户都支持卡片显示/隐藏与排序
- 配置以服务端持久化保存
- 本期不做拖拽布局、不做分栏配置、不做首页顶部视图切换
- 本期不强制新增统一 dashboard 聚合后端接口，先由前端聚合已有协作与报销数据源
